# Doki Doki Development (DDD) 要件定義書

> 心拍数で書いたコードしかコミットできない開発環境

---

# 1. プロジェクト概要

## 1.1 プロダクト名

**Doki Doki Development**  
（略称: DDD / 3D）

---

## 1.2 キャッチコピー

> 「平常心で書いたコードは、信用できない。」

---

## 1.3 コンセプト

開発者の「情熱」を心拍数という生体情報で物理計測し、一定以上の心拍数で書かれたコードのみを正規のコミットとして認める、世界初の生体認証型バージョン管理体験。

「コードの質は熱量に比例する」という新しい開発哲学を提示する。

---

## 1.4 解決したい課題（という名のジョーク）

- 惰性で書かれたコードがリポジトリを汚染する問題
- レビュー時に「なんとなく書いた」コードが紛れ込む問題
- エンジニアの情熱が可視化されない問題
- デモ会場で笑いが取れない問題

---

## 1.5 ターゲットユーザー

- Apple Watch / Fitbit / Garmin などの心拍計測デバイス所有者
- 自分の情熱を定量的に証明したいエンジニア
- 階段ダッシュが好きなエンジニア
- 締切駆動開発の自覚があるエンジニア

---

# 2. 機能要件

## 2.1 コア機能

### F-01: 心拍数連動コミットゲート

#### 概要

`git commit` 実行時に現在の心拍数を取得し、閾値判定を行う。

#### 詳細

- 心拍数が **120 bpm 超** の場合のみコミットを許可
- 平常時のコミットは `PassionInsufficientError` で拒否

#### エラーメッセージ例

```txt
✗ Commit rejected: Heart rate 72 bpm < 120 bpm

Reason: 情熱が足りません。階段を上ってきてください。
```

---

### F-02: 心拍数計測

#### 概要

ウェアラブルデバイスからリアルタイムに心拍数を取得する。

#### 対応デバイス（優先順）

1. Apple Watch（HealthKit）
2. Fitbit（Web API）
3. Garmin Connect IQ
4. Web Bluetooth 対応心拍計

#### 計測頻度

コミット実行時点から過去10秒間の平均値を採用する。

---

### F-03: コード入力ロック（オプション）

#### 概要

心拍数が閾値未満の状態では、エディタ入力も制限する。

#### 対応エディタ

- VS Code 拡張

#### 挙動

- キー入力を間引く
- 自動で以下を挿入する

```cpp
// TODO: もっと熱くなってから書く
```

---

## 2.2 心拍数を上げる手段（公式サポート）

DDD では以下を正式な「心拍数ブースト手段」として認定する。

| 手段 | 期待される心拍上昇 | 副作用 |
|---|---|---|
| 締切ギリギリ駆動 | +40 bpm | 寿命短縮 |
| コーヒー10杯 | +25 bpm | タイポ増加 |
| 上司からの Slack 通知 | +35 bpm | 精神ダメージ |
| 階段ダッシュ | +60 bpm | 物理的疲労 |
| ホラー動画 BGM | +15 bpm | 集中力低下 |

---

## 2.3 応用機能

### F-04: 絶望度メーター

#### 概要

コードの汚さと心拍数の相関を可視化する。

#### 指標

- X軸: 心拍数（bpm）
- Y軸: Cyclomatic Complexity

#### 称号システム

| 条件 | 称号 |
|---|---|
| 高心拍 × 低複雑度 | 真のアスリート開発者 |
| 高心拍 × 高複雑度 | パニックコーダー |
| 低心拍 × 低複雑度 | 冷静な達人 |
| 低心拍 × 高複雑度 | 怠惰な悪魔 |

---

### F-05: PR 自動心拍グラフ添付

#### 概要

Pull Request 作成時、コミット中の心拍推移グラフを自動挿入する。

#### 表示内容

- 時系列心拍グラフ
- 各コミット時点の bpm
- 平均心拍
- 最高心拍
- コミット成功率

#### 実装方式

- GitHub Actions
- GitHub App

---

### F-06: 情熱ランキング

#### 概要

チーム内の「平均コミット時心拍数」をランキング表示する。

#### インセンティブ

月間 MVP には「階段ダッシュ用スニーカー」が贈られる（という設定）。

---

# 3. 非機能要件

## 3.1 性能

| 項目 | 要件 |
|---|---|
| 心拍取得〜判定レイテンシ | 3秒以内 |
| コミット時オーバーヘッド | +5秒以内 |

---

## 3.2 信頼性・セキュリティ

- 心拍データはローカル処理を基本とする
- 外部送信時は HTTPS 必須

### 心拍偽装対策

v1 ではあえて対策しない。

将来的には ECG 波形による本人認証を検討する。

---

## 3.3 倫理的配慮

- 医療機器ではないことを明示
- 心疾患を持つ利用者への注意喚起
- 過度な心拍上昇演出を避ける
- UI 文言はユーモラスに保つ

---

## 3.4 対応プラットフォーム

- macOS（優先）
- Windows
- Linux

構成:

- CLI（git hook）
- VS Code 拡張
- Web ダッシュボード

---

# 4. システム構成

## 4.1 アーキテクチャ概要

```txt
┌─────────────────┐      ┌──────────────────┐      ┌─────────────────┐
│ Apple Watch     │──BLE→│ iPhone           │──API→│ DDDD Daemon     │
│ (HealthKit)     │      │ (Companion App)  │      │ (ローカル常駐)   │
└─────────────────┘      └──────────────────┘      └────────┬────────┘
                                                            │
                         ┌──────────────────────────────────┤
                         │                                  │
                         ▼                                  ▼
                ┌─────────────────┐              ┌──────────────────┐
                │ git pre-commit  │              │ VS Code 拡張     │
                │ hook            │              │ (Status Bar表示) │
                └─────────────────┘              └──────────────────┘
                         │
                         ▼
                ┌─────────────────┐
                │ GitHub App      │
                │ (PR グラフ自動  │
                │ 添付)          │
                └─────────────────┘
```

---

## 4.2 技術スタック（想定）

| レイヤー | 技術 | 理由 |
|---|---|---|
| iOS Companion App | Swift / SwiftUI + HealthKit | Apple Watch 連携 |
| ローカルデーモン | Go / Rust | 軽量・常駐向き |
| Git Hook | Bash | git 統合 |
| VS Code 拡張 | TypeScript | エコシステム |
| ダッシュボード | Flutter Web | 既存スキル |
| バックエンド | FastAPI + Firestore | スピード重視 |
| PR グラフ | GitHub Actions + Chart.js | GitHub 統合 |

---

## 4.3 データフロー

1. Apple Watch が心拍数を計測
2. iPhone に転送（HealthKit）
3. Companion App が DDDD Daemon に送信
4. 開発者が `git commit`
5. hook が心拍数を問い合わせ
6. 閾値判定
7. commit 許可 / 拒否
8. 心拍データ保存
9. PR にグラフ生成

---

# 5. ユーザーストーリー

## US-01: 通常コミット

### ストーリー

開発者として、自分の情熱が十分か知りたい。

### 受け入れ条件

- bpm が表示される
- 120 bpm 超なら commit 成功
- 未満なら拒否

---

## US-02: 階段ダッシュコミット

### ストーリー

発表中に階段を駆け上がり、その勢いでコミットしたい。

### 受け入れ条件

- 140 bpm 以上
- 派手な演出
- 以下を表示

```txt
🔥 PASSION COMMIT ACCEPTED 🔥
```

---

## US-03: PR レビュー

### ストーリー

レビュアーとして「どれだけ情熱で書かれたか」を知りたい。

### 受け入れ条件

- 心拍グラフ自動添付
- 情熱不足ラベル付与
