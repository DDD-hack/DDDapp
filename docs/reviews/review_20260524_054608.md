## 🟡 Minor Issues

### [m-1] http.Client を毎回生成している

**ファイル**: `daemon/internal/discord/webhook.go:49`  
**問題**: `Send` が呼ばれるたびに `http.Client` を新規作成する。内部の接続プールが使い捨てになり、TCP コネクション再利用が効かない。コミットごとに1回の頻度なら実害はないが、慣習に反する。

```go
// ❌ 現在（毎回生成）
c := &http.Client{Timeout: 5 * time.Second}
resp, err := c.Do(req)
```

```go
// ✅ パッケージ変数にする
var httpClient = &http.Client{Timeout: 5 * time.Second}

func Send(ctx context.Context, webhookURL string, p Payload) error {
    ...
    resp, err := httpClient.Do(req)
```

### [m-2] レスポンスボディを読み捨てていない

**ファイル**: `daemon/internal/discord/webhook.go:54`  
**問題**: `defer resp.Body.Close()` のみで、ボディを読まずに閉じると HTTP/1.1 の Keep-Alive が機能しない（接続が再利用されない）。Discord は 204 No Content を返すのでボディは空だが、明示的に読み捨てるのが慣習。

```go
// ❌ 現在
defer resp.Body.Close()
```

```go
// ✅ 修正案
defer func() {
    _, _ = io.Copy(io.Discard, resp.Body)
    resp.Body.Close()
}()
```

## 🟢 Good Points

### [G-1] SetDiscordWebhookURL の書き込みはサーバー起動前に完結している
**ファイル**: `daemon/cmd/ddd/main.go:74-75`  
`SetDiscordWebhookURL` は `e.Start()` より前に呼ばれ、以降は読み取り専用。Go のメモリモデル上 happens-before が成立しており、Mutex なしで安全。

### [G-2] goroutine に値コピーを渡している
**ファイル**: `daemon/internal/api/handler.go:221-228`  
RTDB 書き込み goroutine はリクエスト変数をクロージャではなく引数コピーで受け取っており、goroutine リークや data race を防いでいる。

### [G-3] Send が空 URL で即時 nil を返す
**ファイル**: `daemon/internal/discord/webhook.go:36-38`  
webhook 未設定時の no-op が呼び出し側でなく Send 内で処理されており、呼び出し側のコードがシンプルになっている。

### [G-4] commitPayload が純粋関数になっている
**ファイル**: `daemon/internal/api/handler.go:379-431`  
副作用なし・状態参照なし。単体テストが書きやすく、将来のメッセージフォーマット変更も安全。

### [G-5] goroutine に 5 秒タイムアウトを設定している
**ファイル**: `daemon/internal/api/handler.go:237-243`  
Discord API が応答しなくても goroutine がリークしない。`ctx, cancel` の `defer cancel()` も適切。
