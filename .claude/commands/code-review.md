# Code Review Skill

指定されたコードを多角的にレビューし、**指摘事項レポートをMarkdown形式で出力**する。

## 入力

`$ARGUMENTS` にファイルパス、ディレクトリ、または機能名を受け取る。
未指定の場合は `git diff` または直近の変更ファイルを対象にする。

## 実行フロー

### Step 1: レビュー対象の特定

```bash
# 引数がある場合
find $ARGUMENTS -type f \( -name "*.go" -o -name "*.ts" -o -name "*.tsx" -o -name "*.swift" \) 2>/dev/null

# 引数がない場合: 直近のgit変更を取得
git diff --name-only HEAD 2>/dev/null || git diff --cached --name-only 2>/dev/null

# ステージング済みの変更
git diff --staged --name-only 2>/dev/null
```

### Step 2: コンテキスト収集

レビュー前にプロジェクトの規約を把握する:

```bash
# プロジェクト規約の確認
cat CLAUDE.md 2>/dev/null

# Go lintルールの確認
cat .golangci.yml 2>/dev/null

# 既存テストの確認
find . -name "*_test.go" | head -20
```

### Step 3: 多角的レビューの実施

以下の観点で各ファイルを分析する。**必ず実際のコードを引用し、架空のコードは書かない。**

#### 🔴 Critical（必ず修正）
- **バグ・ロジックエラー**: 条件分岐の誤り、null安全性の欠落
- **セキュリティ問題**: シークレットのハードコード、認証バイパス
- **データ損失リスク**: 例外未処理、トランザクション漏れ
- **race condition**: Mutex なしの並行アクセス

#### 🟠 Major（早めに修正）
- **パフォーマンス問題**: 不要なアロケーション、メモリリーク
- **設計の問題**: CLAUDE.md の規約違反、責任の分離違反
- **エラーハンドリング**: エラーの握りつぶし（`_ = err` など）

#### 🟡 Minor（改善推奨）
- **命名**: 意図が伝わらない変数名・関数名
- **可読性**: 複雑すぎる条件式、深すぎるネスト
- **コメント**: 不要なコメント、必要なのにないコメント

#### 🟢 Good（良い点）
- うまく実装されている箇所を最低3つ記載する

#### 🔵 Suggestion（提案）
- リファクタリングの機会（コード例付き）

#### 📝 Testing
- テストが不足している箇所とテストケースの提案

### Step 4: DDD プロジェクト固有チェック

CLAUDE.md の規約に照らして以下を必ず確認する:

- [ ] `os.UserHomeDir()` を使っているか（`~` を使っていないか）
- [ ] `filepath.Join()` でパスを結合しているか
- [ ] `log.Fatal` が main.go 以外で使われていないか
- [ ] エラーを必ず返しているか（握りつぶしていないか）
- [ ] 並行アクセスに `sync.Mutex` があるか
- [ ] `mattn/go-sqlite3` を使っていないか（`modernc.org/sqlite` を使うこと）
- [ ] `GOOS=windows go build` が通るか（CGO を使っていないか）
- [ ] WebSocket の `defer conn.Close()` があるか
- [ ] daemon 未起動時に `exit 0` になっているか（`exit 1` にしない）

### Step 5: レビューレポートの生成

`docs/reviews/review_{YYYYMMDD_HHMMSS}.md` に出力する:
